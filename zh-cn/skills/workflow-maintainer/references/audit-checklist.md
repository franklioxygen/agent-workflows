# 工作流维护审计清单

## 阻塞检查

- 必需的工作流文件存在，且命名一致。
- 必需的共享文件存在：`repository-preflight.md`、`safety-rules.md` 和 `workflow-conventions.md`。
- Markdown 链接和标题锚点都能正确解析。
- README 清单列出了所有工作流文件、共享文件和内置技能。
- 工作流路由和库加载参考与实际文件名一致。
- 自动化脚本不会为不完整的库返回成功结果。
- 技能目录中的 `SKILL.md` frontmatter 有效，且 `name` 与目录名匹配。
- 每个可安装技能都只有一个规范的代理元数据文件：`agents/interface.yaml`。
- 不存在遗留的按代理拆分的元数据文件，例如 `agents/openai.yaml` 和 `agents/claude.yaml`。
- 技能代理默认提示词中提到了字面量 `$skill-name`。

## 设计检查

- 工作流专属内容没有不必要地重复共享规则。
- 安全相关措辞在提示词和共享规则之间保持一致。
- 分诊门能在小、中、高风险工作之间正确路由，而不会给出互相矛盾的后续步骤。
- 只读评审类工作流不会在用户未明确要求修复时暗示要进行编辑。
- 事故工作流会把破坏性动作、部署和缓解措施都放在明确批准之后。
- 重构和清理工作流在需要交接之前都保持可观察行为不变。

## 交接检查

- 如果某个工作流会跨边界，文档要说明何时暂停、由哪个工作流接手。
- 最终报告包含验证命令、跳过验证的原因，以及提交/推送确认。
- 评审和清理工作流要说明本轮是否改了代码，还是保持只读。
- 验证交接会把既有失败与当前任务引入的回归区分开。
- 后续工作明确、范围清晰，并与已完成任务分离。
- 任何尚未解决的歧义都被明确写出，而不是被隐藏成默认假设。
