# 缺陷修复代理工作流

本文定义了一套可复用的代理工作流，用于诊断并修复缺陷。输入可以是缺陷描述、GitHub issue 链接或错误日志。

该工作流面向在仓库中工作的 AI 编码代理。与功能开发工作流相比，它更轻量，因为缺陷修复的范围更窄：理解问题、找到根因、修复问题，并验证修复结果。

## 预检仓库检查

在进入分诊门之前，使用 [标准编码预检](shared/repository-preflight.md#standard-coding-preflight)。

## 全局规则

在每个提示词中使用 [标准编码规则](shared/safety-rules.md#standard-coding-rule)。

## 分诊门

开始之前，先对缺陷分类，以选择合适的流程深度：

```text
将这个缺陷归类到以下类别之一：

1. Obvious：从描述中就能看出根因。修复是小而局部的改动。例子：错字、错误常量、缺少 null 检查、off-by-one 错误、错误配置值。

2. Moderate：症状很清楚，但根因需要调查。修复可能涉及 1 到 3 个文件。例子：错误的查询逻辑、缺失校验、错误事件处理器、已知触发条件下的竞态问题。

3. Complex：症状模糊或间歇出现，根因不清楚，或修复会有跨领域影响。例子：数据损坏、间歇性失败、性能下降、涉及多个服务或子系统的问题。

根据分类：

- Obvious：直接使用 Step 2（Fix），并在输入中内联根因和失败复现。在满足下面“复现要求”之前，不要编辑代码。
- Moderate：从 Step 1（Diagnose）开始。让调查保持聚焦。
- Complex：完整执行 Step 0 到 Step 4。

缺陷报告：

<缺陷描述、GitHub issue 链接或错误日志>

请对这个缺陷分类，并建议应遵循哪些步骤。
```

## 复现要求

在编辑代码之前，先建立以下任一项：

- 一个能捕获该缺陷的失败自动化测试
- 一个能复现该缺陷的失败命令或脚本
- 清晰的手工复现步骤，并附带当前失败证据
- 对“为什么无法直接复现”的明确解释（写明具体技术原因），以及当前能拿到的最佳证据。“无法复现”意味着该缺陷依赖生产数据、第三方状态或本地无法模拟的基础设施，而不是“复现起来比较麻烦”

后续工作流默认这个复现凭证已经存在，并且在验证阶段可以重复使用。

## Step 0：理解缺陷

当缺陷来自 GitHub issue 链接或描述比较模糊时，使用下面提示词：

```text
阅读下面的缺陷报告，并在现有代码库中做足够的检查来理解问题。

缺陷报告：

<缺陷描述、GitHub issue 链接或错误日志>

你的目标：

1. 总结上报行为和期望行为。
2. 识别受影响的代码区域（文件、模块、端点、UI 组件）。
3. 确定复现路径：如果报告中有描述则提取；如果没有但可以推断，也请说明。
4. 将结果拆分为：
   - Confirmed facts（从报告和代码库中已知的事实）
   - Hypotheses（可能的根因，按概率排序）
   - Missing information（确认根因还需要什么信息）
5. 如果缺陷报告是 GitHub issue 链接，还要额外提取：
   - 能补充上下文的相关评论、关联 PR、标签或负责人
   - 报告者提供的复现步骤

先不要修这个缺陷。
未经我明确许可，不要提交或推送。

如果缺陷已经清楚到足以进入诊断，请明确说明，并给出按优先级排序的假设。
如果缺少关键信息，请列出继续推进所需的具体问题。
```

## Step 1：诊断根因

在理解缺陷之后，使用下面提示词：

```text
在代码库中调查以下缺陷的根因。

缺陷摘要：

<来自 Step 0 的已确认行为、期望行为、受影响区域和假设；若跳过 Step 0，则填写清晰的缺陷描述>

你的目标：

1. 在可行时，用失败测试、失败命令或成文的手工证据复现问题。
2. 追踪与该行为相关的代码路径。
3. 找出精确根因。不要猜测，要沿着代码走到结论。
4. 解释为什么当前代码会产生错误行为。
5. 判断根因属于哪类：
   - 逻辑错误
   - 缺失检查或校验
   - 数据问题（错误输入、陈旧状态、迁移缺口）
   - 竞态或时序问题
   - 依赖或集成问题
   - 配置问题
6. 找出需要改动的最小文件集合。
7. 检查该缺陷是否在别处也有相关症状（同样模式是否在其他地方重复出现、同一函数的其他调用方、存在相同缺陷的共享工具函数）。
8. 评估影响面：修复是否可能破坏现有行为、其他调用方或下游使用方？

先不要开始修复。
未经我明确许可，不要提交或推送。

请提供：
- 复现凭证（或为什么无法直接复现的解释）
- 根因说明
- 涉及文件
- 影响面评估
- 建议的修复方式
- 修复必须处理的风险或边界场景
```

## Step 2：修复缺陷

在确认根因之后，使用下面提示词：

```text
根据下面的诊断结果修复这个缺陷。

缺陷摘要：

<来自 Step 1 的根因说明、涉及文件、建议修复方案和风险；若跳过前置步骤，则填写清晰描述>

重要说明：

1. 编辑前仔细阅读相关代码。
2. 在修改代码前确认已经满足“复现要求”。如果无法直接复现，请说明原因并使用当前最佳证据。
3. 遵循代码库现有模式。
4. 让修改严格限定在缺陷修复范围内。不要顺手重构、清理或改进周边代码。
5. 修复根因，而不是症状。
6. 如果同样的缺陷模式存在于诊断阶段识别出的相关代码路径中，也一并修复，但前提是它们确实共享同一个根因。
7. 添加或更新测试，覆盖：
   - 触发该缺陷的精确场景
   - 与修复相关的边界场景
   - 回归防护（没有修复时测试应失败，修复后应通过）
8. 在可用时运行相关测试、lint 和类型检查。
9. 未经我明确许可，不要提交或推送。
10. 如果发现真正根因与诊断不同，先停下来解释，再继续。

结束时请提供：

1. 根因（一句话）
2. 改了什么以及为什么
3. 变更文件
4. 修复前使用的复现方式
5. 新增或更新的测试
6. 运行过的命令及结果
7. 本次修复没有覆盖的内容（已知限制、相关问题）
```

## Step 3：验证修复

应用修复后，使用下面提示词：

```text
验证当前工作区里已应用的缺陷修复。

缺陷摘要：

<原始缺陷描述>

根因：

<根因说明>

你的目标：

1. 验证修复的是根因，而不只是症状。
2. 检查修复不会在相关代码路径中引入回归。
3. 审查变更文件的正确性、边界场景和非预期副作用。
4. 验证测试确实覆盖缺陷场景，并且在没有修复时会失败。
5. 运行相关验证命令（测试、lint、类型检查）。
6. 检查是否修改了无关文件。
7. 重新运行工作流早先建立的复现凭证，确认它现在通过或不再可复现。

如果发现问题，请按严重级别分类：

- Critical：合并前必须修复
- Major：合并前应该修复
- Minor：改进项或清理项

每个发现都使用以下格式：

### [Severity] Finding title

- Location:
- Problem:
- Why it matters:
- Recommended fix:
- Status: Open / Fixed

未经我明确许可，不要提交或推送。

如果修复正确且完整，请明确说明，并列出执行过的验证。
```

如有需要，在没有 Critical 或 Major 问题之前，反复执行 Step 3 和 Step 2。

推荐停止条件：

```text
如果 Step 3 发现问题，则回到 Step 2 修复，再重新执行 Step 3。当某次验证通过且没有 Critical 或 Major 问题时，停止循环。
```

## Step 4：最终报告

对于复杂缺陷，或在 Step 3 清理完毕后需要交接总结时，使用下面提示词：

```text
为以下缺陷输出最终修复报告：

缺陷摘要：

<原始缺陷描述或 GitHub issue 链接>

报告应包含：

1. 缺陷摘要（上报行为 vs. 期望行为）
2. 根因
3. 修复说明
4. 变更文件
5. 新增或更新的测试
6. 运行过的命令及结果
7. 影响面评估（还有什么可能受影响）
8. 已知限制或后续事项
9. 确认未执行提交或推送

未经我明确许可，不要提交或推送。
```

## 实务说明

- 另外也要遵循 [shared/workflow-conventions.md](shared/workflow-conventions.md) 中的共享约定。
- 使用分诊门，避免对简单缺陷做过度流程化处理。
- 编辑代码前先建立复现凭证。如果无法直接复现，要明确说明原因，并记录现有最佳证据。
- 修复根因，而不是症状。如果症状消失了，但底层逻辑仍然错误，缺陷还会回来。
- 缺陷修复不是重构机会。
- 如果同样的缺陷模式在别处重复出现，只在它们共享同一根因时一起修复。否则，把它们作为单独问题处理。
- 如果诊断结果表明这是设计缺陷而不是代码 bug，就升级到功能开发工作流，而不是打补丁绕过去。
