# 事故 / 生产排障代理工作流

本文定义了一套可复用的代理工作流，用于诊断和解决生产事故。输入可以是告警、错误日志、监控面板观察结果、用户报告，或 PagerDuty/Slack 升级消息。

该工作流面向在生产事故期间或事故后、在仓库中工作的 AI 编码代理。它强调 **先缓解，后根因**。目标是先止血，然后理解为什么会发生，再正确修复。

该工作流与缺陷修复工作流在紧急程度和处理方式上不同：

- 缺陷修复工作流：按部就班地诊断，然后修复，再验证。
- 事故工作流：先立即缓解影响，然后诊断，再应用正式修复，或交接给缺陷修复工作流。

## 预检仓库检查

在进入分诊门之前，使用 [事故响应预检](shared/repository-preflight.md#incident-response-preflight)。

## 全局规则

在每个提示词中使用 [事故响应规则](shared/safety-rules.md#incident-response-rule)。

## 分诊门

开始之前，先对事故分类，以确定紧急程度：

```text
将这次事故归类到以下类别之一：

1. Active：系统当前处于受损状态。用户正在受影响。例子：服务宕机、错误暴涨、数据正在损坏、支付失败、认证失效。

2. Degraded：系统仍部分可用，但有问题。影响有限，或正在缓慢扩大。例子：延迟升高、一部分用户出现间歇性错误、仅一个区域受影响、后台任务失败、非关键功能损坏。

3. Post-incident：即时影响已经解决或缓解，但根因尚未搞清。例子：服务重启后恢复、回滚止住错误、人工修复已实施但底层问题仍在。

根据分类：

- Active：从 Step 0（Assess and Mitigate）开始。速度优先。先止住影响，再诊断根因。
- Degraded：同样从 Step 0（Assess and Mitigate）开始，但在行动前多花一些时间理解范围。错误的缓解措施可能让退化系统更糟。
- Post-incident：跳到 Step 2（Root Cause Analysis）。系统已经稳定，重点是理解发生了什么。

事故报告：

<告警、错误日志、监控观察结果、用户报告或升级消息>

请对这次事故分类，并建议应遵循哪些步骤。
```

## Step 0：评估并缓解

当系统正在受损或处于退化状态时，使用下面提示词：

```text
评估这次生产事故，并识别缓解选项。

事故报告：

<告警、错误日志、监控观察结果、用户报告或升级消息>

### Part A：快速评估

你的目标：

1. 确定出了什么问题：受影响的是哪个服务、端点、功能或数据路径。
2. 确定影响面：有多少用户、请求或操作受到影响。
3. 确定时间线：什么时候开始的？是在恶化、持平，还是好转？
4. 检查与事故相关的近期变更：
   - 最近部署（git log、部署历史、CI/CD）
   - 最近配置变更
   - 最近数据库迁移
   - 最近依赖升级
   - 最近基础设施变更
5. 检查外部因素：
   - 第三方服务故障
   - 流量突增
   - 计划内维护
6. 确定哪些部分仍然正常：系统中未受影响的部分是什么。这有助于缩小排查范围。

### Part B：缓解选项

基于评估结果，按速度和安全性排序提出缓解方案：

对每个方案，提供：
- Action：做什么
- Expected effect：它会修复什么
- Risk：可能出什么问题
- Reversibility：是否容易撤销
- Time to effect：多快能见效

常见缓解模式（优先按以下顺序考虑）：
1. 回滚到最后一个已知正常的部署
2. 关闭某个特性开关
3. 扩容资源
4. 阻断或限流问题流量
5. 重启受影响服务
6. 应用定向热修
7. 切换到备用系统
8. 人工数据修正

未经我明确许可，不要执行任何缓解措施。
未经我明确许可，不要提交、推送或部署。
未经我明确许可，不要运行破坏性命令。

请提供：
- 什么坏了，以及从什么时候开始
- 影响面（用户、请求、收入、数据完整性）
- 最可能触发因素（近期变更、外部因素或未知）
- 按优先级排序的缓解选项
- 继续推进还需要我提供什么
```

## Step 1：稳定系统并收集证据

在缓解已应用之后，或系统处于退化但正在监控时，使用下面提示词：

```text
以下缓解措施已经应用（或系统正在被监控）：

已应用的缓解措施：

<已执行的动作，或“none yet — monitoring”>

当前状态：

<当前系统状态：已恢复、部分恢复、仍然退化>

你的目标：

1. 确认缓解是否有效：
   - 错误率是否在下降？
   - 受影响用户是否在恢复？
   - 缓解措施是否带来了新的症状或副作用？
2. 趁事故记忆仍然新鲜，收集根因分析所需证据：
   - 相关日志条目（错误、警告、异常模式）
   - 堆栈跟踪或错误信息
   - 能把症状与变更或事件关联起来的时间戳
   - 受影响的请求路径、用户分群或数据子集
   - 监控数据（延迟、错误率、吞吐、资源使用）
3. 识别缓解措施没有解决的问题：仍然存在的残余问题、数据不一致或功能退化。
4. 判断缓解是临时性的（仍需正式修复）还是永久性的（问题已被完全解决）。

未经我明确许可，不要提交、推送或部署。

请提供：
- 缓解效果（working / partially working / not working）
- 已收集证据（日志、trace、指标、时间戳）
- 残余问题
- 是否仍需要正式修复
- 建议的下一步：根因分析、进一步缓解或升级处理
```

## Step 2：根因分析

系统稳定后，使用下面提示词：

```text
对下面这次生产事故做根因分析。

事故摘要：

<发生了什么、何时发生、影响面以及已应用的缓解措施>

证据：

<来自 Step 1 的日志、trace、指标、时间戳和观察结果；若跳过前置步骤，则使用事故报告中的信息>

你的目标：

1. 从症状追溯到根因。跟着证据走，不要靠假设。
2. 建立因果链：
   - Trigger：是什么触发了故障（部署、配置变更、流量激增、数据条件、外部依赖）
   - Mechanism：触发因素如何导致症状（代码路径、数据流、资源耗尽、竞态）
   - Impact：到底坏了什么，以及为什么它之前没有被发现
3. 识别促成因素：
   - 缺失监控或告警
   - 缺失输入校验或错误处理
   - 缺失本应能发现问题的测试
   - 部署或回滚流程缺口
   - 文档或运行手册缺口
4. 判断根因类别：
   - 代码缺陷
   - 配置错误
   - 数据完整性问题
   - 基础设施 / 容量问题
   - 外部依赖故障
   - 流程 / 人为错误
5. 评估当前缓解是否真正解决问题，还是只是掩盖了问题。
6. 检查相关风险：相同根因是否可能在其他地方以不同形式再次出现？

未经我明确许可，不要提交、推送或部署。

请提供：
- 根因（一段话）
- 因果链（trigger → mechanism → impact）
- 促成因素
- 根因类别
- 当前缓解是否足够，还是仍需正式代码修复
- 相关风险
```

## Step 3：应用正式修复

当根因已经清楚，且需要代码修复时，使用下面提示词：

```text
基于下面的根因分析，为这次生产事故应用正式修复。

根因：

<来自 Step 2 的根因说明、因果链和相关风险>

当前生效的缓解措施：

<当前启用的临时缓解措施（如果有）>

重要说明：

1. 编辑前仔细阅读相关代码。
2. 修复根因，而不是症状。缓解措施只负责止血；正式修复必须防止问题再次发生。
3. 遵循代码库现有模式。
4. 让改动严格限定在此次事故范围内。不要顺手重构、清理或改进周边代码。
5. 如果根因分析表明同一根因也影响其他代码路径，也一并修复。
6. 添加或更新测试，覆盖：
   - 导致事故的精确场景
   - 与根因相关的边界场景
   - 回归防护（没有修复时测试应失败）
7. 如果当前有临时缓解措施（特性开关、配置覆盖、回滚），说明在本次修复部署后是否可以移除。
8. 运行相关测试、lint 和类型检查。
9. 未经我明确许可，不要提交、推送或部署。
10. 如果正式修复对于热修来说风险太高或过于复杂，请明确指出，并建议一个更安全的短期修复，以及后续的完整修复方案。

结束时请提供：

1. 根因（一句话）
2. 改了什么以及为什么
3. 变更文件
4. 新增或更新的测试
5. 运行过的命令及结果
6. 部署后是否可以移除临时缓解措施
7. 本次修复没有覆盖的内容（后续事项、相关风险）
```

## Step 4：事故后报告

当事故已经完全解决后，使用下面提示词：

```text
为下面这次生产事故输出事故后报告。

事故摘要：

<发生了什么、何时发生、影响面、缓解措施、根因以及已应用修复>

报告应包含：

1. Incident Timeline
   - 事故开始时间（或首次被发现时间）
   - 团队收到告警的时间
   - 缓解措施应用时间
   - 系统恢复时间
   - 正式修复部署时间（如适用）
   - 用户受影响总时长

2. Impact Summary
   - 受影响的用户、请求或操作
   - 收入或数据影响（如果已知）
   - SLA 或 SLO 影响（如果适用）

3. Root Cause
   - Trigger
   - Mechanism
   - Why it was not caught earlier

4. Mitigation and Fix
   - 已应用的缓解措施及其效果
   - 已应用的正式修复（或计划中的修复）
   - 变更文件
   - 新增测试

5. Contributing Factors
   - 除直接根因外，让事故成为可能的因素
   - 流程、工具、监控或测试上的缺口

6. 行动项
   对每个 action item，提供：
   - Description
   - Priority: P0 (immediate) / P1 (this sprint) / P2 (next sprint) / P3 (backlog)
   - Owner（如果未知就留空，不要自行指派）
   - Status: Open / In Progress / Done

   常见 action item 类别：
   - 代码修复（如果尚未部署）
   - 监控和告警改进
   - 测试覆盖补充
   - 运行手册创建或更新
   - 流程改进
   - 文档更新

7. 经验总结
   - 响应过程中做得好的地方
   - 还可以改进的地方

未经我明确许可，不要提交、推送或部署。
```

## 实务说明

- 另外也要遵循 [shared/workflow-conventions.md](shared/workflow-conventions.md) 中的共享约定。
- 先缓解，再诊断。2 分钟内通过回滚止住错误，要比用户持续受影响 30 分钟却只完成一份根因分析更好。缓解速度和诊断完整性是两条不同优先级，不要为了其中一项牺牲另一项。
- 未经明确许可，不要执行破坏性命令。这包括数据库变更、强推、服务重启和缓存清空。先说明你想做什么，等待批准。
- 趁证据还没消失时先保存下来。日志会轮转、指标会过期、重启后临时状态会消失。要尽早抓取相关日志、trace 和指标。
- 把缓解和修复分开。回滚或关闭特性开关是缓解，不是修复。底层的代码缺陷、配置错误或数据问题仍然需要处理。两者都要跟踪。
- 如果根因分析表明这是设计缺陷，而不是代码 bug，就升级到功能开发工作流进行正式重设计。不要在事故响应中给架构问题打补丁。
- 如果正式修复对热修来说风险太高，就先应用安全的短期修复，并把完整修复作为后续事项跟踪。因为仓促修复导致第二次事故，比带着已知限制和明确时间线更糟。
- 没有负责人的 action item 会自然腐烂。你可以标出未指定负责人的事项，但不要替团队分派负责人。
- 如果你不确定某个动作是否安全，先问。事故中，提问的成本只是几秒；做错动作的成本可能是几小时。
