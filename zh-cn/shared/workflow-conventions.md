# 共享工作流约定

说明：`agent-workflows` 工作流文档复用的跨工作流执行约定。

这些是多个工作流里都会反复出现的共性要求。把工作流专属步骤保留在各自文档中，把共享执行规则放在这里。

## 工作区安全

- 在分诊前先运行相关预检，以便尽早掌握仓库说明、工作区状态和无关本地变更。
- 不要覆盖、回退或重新格式化与当前任务无关的用户变更。
- 让每个工作单元都足够小，便于独立评审、验证和回退。

## 范围控制与升级

- 让修改严格限定在当前任务或评审目标内。
- 如果任务比预期更大、风险更高或歧义更多，先停下来重新分类、重新规划，或先询问后再继续。
- 如果工作需要设计决策、行为变更或无关清理，不要在当前工作流里临时处理，切换到更合适的工作流。

## 验证与基线

- 对于会改代码的工作流，在可行时先建立并记录基线，这样才能把回归与既有失败区分开。
- 每完成一个逻辑单元后重新运行计划中的验证集，并在结束时再完整运行一次。
- 使用与任务真正相关的完整验证集，而不仅仅是测试：还包括类型检查、lint、构建或打包检查、使用情况检查、冒烟测试、审计或定向运行时检查。

## 报告与交接

- 始终说明运行了哪些命令或验证步骤，哪些通过、哪些失败、哪些被跳过以及原因。
- 把既有失败与当前工作引入的失败明确分开。
- 如果工作流里有后续验证或复评步骤，就按要求在执行/修复步骤与验证/复评步骤之间循环，直到阻塞问题解决，或工作流明确允许停止。

## 评审质量

- 优先给出具体、有证据支撑的发现，而不是泛泛而谈。
- 如果问题属于设计层面，要尽早指出，不要把它淹没在后面可能失去意义的实现细节里。
- 对评审和验证步骤中的所有发现都使用以下结构化格式：

```
### [Severity] Finding title

- Location:
- Problem:
- Why it matters:
- Recommended fix:
- Status: Open / Fixed / Needs decision
```

严重级别定义：

- Critical：会导致数据丢失、安全漏洞、崩溃或功能损坏。合并前必须修复。
- Major：行为不正确、缺少边界条件处理或存在重大风险。合并前应该修复。
- Minor：代码质量、可维护性或轻微正确性问题。建议修复，但不阻塞。
- Nit：风格、命名或偏好问题。可选。

## 工作流交接

- 一次只使用一个主工作流。不要在同一轮里混用多个工作流的步骤。
- 如果任务跨越多个工作流边界（例如缺陷修复暴露出设计缺陷，或者清理任务需要行为变更），先完成或暂停当前工作流步骤，再明确交接到下一个工作流。
- 切换时要说明交接原因：发现了什么、为什么当前工作流不再合适、接下来该继续哪个工作流。
- 常见交接路径：
  - Incident -> Bug Fix：完成缓解并保存证据后，如果剩余工作是常规代码修复，则切到缺陷修复工作流。
  - Bug Fix -> Feature：如果诊断发现问题是设计或产品决策，而不是狭义缺陷，则切到功能工作流。
  - Tech Debt -> Refactoring：如果清理工作收敛成针对某个局部区域的、保持行为不变的结构调整，则切到重构工作流。
  - Tech Debt 或 Refactoring -> Feature：如果工作需要行为变更、公共契约变更或设计决策，则切到功能工作流。
  - Code Review -> Fix workflow：如果用户明确要求处理发现项，并且这些发现对应功能、缺陷或重构任务，则切到相应的修复工作流。
