# 重构代理工作流

本文定义了一套可复用的代理工作流，用于在不改变行为的前提下改进代码结构。输入可以是明确的重构目标、某种代码异味、一个技术债项，或一个需要重新组织的模块。

该工作流面向在仓库中工作的 AI 编码代理。它的核心约束是：**行为不能改变**。每一步都服务于一个目标：确保重构能改善代码，同时保留已有功能。

## 预检仓库检查

在进入分诊门之前，使用 [标准编码预检](shared/repository-preflight.md#standard-coding-preflight)。

## 全局规则

在每个提示词中使用 [保持行为不变规则](shared/safety-rules.md#behavior-preserving-rule)。

## 分诊门

开始之前，对重构进行分类，以选择合适的流程深度：

```text
将这次重构归类到以下类别之一：

1. Safe：机械性改动，没有行为风险。现有测试或类型检查已经完整覆盖。例子：重命名变量/函数/文件、抽取局部 helper、删除死代码、重排 import、统一格式。

2. Moderate：模块内或少量文件内的结构调整。行为应该不变，但变换并不简单。例子：抽取类/模块、内联抽象、用多态替换条件分支、合并重复逻辑、在已知调用方范围内修改函数签名。

3. Architectural：跨领域的结构性变更，会影响模块边界、数据流、依赖图或公共接口。例子：拆分巨型模块、修改共享数据结构、在整个代码库里替换一种模式、迁移到新抽象、重组目录布局。

根据分类：

- Safe：从 Step 0（建立基线）开始。如果基线确认改动确实低风险，则直接进入 Step 3（执行）。除非工作中风险上升，一次执行/验证循环通常就够。
- Moderate：从 Step 0（建立基线）开始，然后进入 Step 2（规划）。如果范围、依赖或接口还不清楚，就先执行 Step 1（理解当前状态）。
- Architectural：完整执行 Step 0 到 Step 5。

重构目标：

<要重构什么，以及为什么>

请对这次重构分类，并建议应遵循哪些步骤。
```

## Step 0：建立基线

对任何重构（包括 Safe 类别），在编辑代码前使用下面提示词：

```text
在开始重构前建立验证基线。

重构目标：

<要重构什么，以及为什么>

你的目标：

1. 识别与受影响代码相关的验证命令：
   - Tests
   - Type checks
   - Linters or format checks
   - Build/package commands
   - Manual smoke tests or targeted runtime checks
2. 现在就运行完整、相关的验证集合，并记录结果。
3. 将结果拆分为：
   - Passing baseline checks
   - Pre-existing failures
   - Missing coverage or validation gaps
4. 判断以当前安全网是否足以开始重构。
5. 如果验证覆盖太弱，建议是否应该：
   - 先补测试
   - 缩小重构范围
   - 升级到更高风险路径
6. 如果重构看起来比最初分类更广或更危险，在做任何编辑前明确指出。

现在先不要编辑代码。
未经我明确许可，不要提交或推送。

请提供：
- 重构过程中要用的验证命令
- 已通过的基线检查
- 需要关注的既有失败
- 覆盖缺口与风险
- 建议的下一步
```

工作流后续步骤都默认这份基线已经记录，并会在每一轮执行后拿来对比。

## Step 1：理解当前状态

对于 Architectural 重构，或者当范围还不清楚时，使用下面提示词：

```text
在规划重构前，先理解需要修改的代码当前状态。

重构目标：

<要重构什么，以及为什么>

你的目标：

1. 识别会被这次重构影响的文件、模块、类和函数。
2. 绘制依赖图：哪些代码依赖这部分代码，这部分代码又依赖什么。
3. 识别必须保持稳定的公共接口（API、导出、共享类型、数据库契约）。
4. 评估受影响代码的现有测试覆盖：
   - 哪些行为已有测试？
   - 哪些行为还没有测试？
   - 现有测试是在测行为（适合围绕它重构），还是在测实现细节（重构过程中可能被合理打破）？
5. 识别约束条件：
   - 向后兼容要求
   - 性能敏感路径
   - 代码所有权或审批要求
   - 下游使用方（其他服务、包或团队）
6. 判断这次重构能否渐进式进行，还是必须一次性协调改完。

现在先不要编辑代码。
未经我明确许可，不要提交或推送。

请提供：
- 受影响的文件和模块
- 依赖图（谁依赖它，它依赖谁）
- 必须保持稳定的公共接口
- 测试覆盖评估
- 约束与风险
- 这次重构是渐进式还是协调式
```

## Step 2：规划重构

在理解当前状态之后（或者范围清晰的 Moderate 重构可直接在 Step 0 之后），使用下面提示词：

```text
为下面这次变更创建重构计划。

重构目标：

<要重构什么，以及为什么>

当前状态：

<来自 Step 1 的受影响文件、依赖图、公共接口、测试覆盖和约束；若跳过 Step 1，则填写简短描述>

基线：

<来自 Step 0 的验证命令、基线结果和覆盖缺口>

计划应包含：

1. 按顺序排列的变换步骤。每一步都应当：
   - 小到可以独立验证
   - 自身保持行为不变（每一步之后代码都应可工作，而不是只有全部完成后才可工作）
   - 用具体动作描述（例如“从 Y 中抽取函数 X”“把文件 A 移到 B 并更新 import”），而不是笼统目标
2. 每一步之后要运行的验证（测试集、类型检查、linter、手工冒烟）。
3. 必须始终保持稳定的公共接口。
4. 哪些测试可能需要改变，因为它们测试的是实现细节而不是行为。要把它们与不该改变的测试区分开（行为测试一旦失败，就意味着重构引入了回归）。
5. 在重构前或重构过程中，为增加信心而需要新增的测试。
6. 回滚策略：如果进行到一半出问题，如何撤销重构。

现在先不要编辑代码。
未经我明确许可，不要提交或推送。

如果计划暴露出这次重构比预期更危险，请明确说明，并建议应继续、缩小范围，还是先补测试。
```

## Step 3：执行重构

当计划明确后（或者 Safe 重构可直接在 Step 0 之后），使用下面提示词：

```text
执行下面的重构。

重构目标：

<要重构什么，以及为什么>

计划：

<来自 Step 2 的有序变换步骤；对于 Safe 重构，可填写简短说明>

基线：

<来自 Step 0 的验证命令、基线结果和覆盖缺口>

重要说明：

1. 在做任何修改前确认基线已被记录。基线中可以包含既有失败，但必须记录清楚，避免与这次重构引入的回归混淆。
2. 如果基线表明覆盖缺失、接口不稳定，或依赖影响比预期更广，先停下来，升级回 Step 1 或 Step 2 再继续。
3. 一次执行一个变换步骤。
4. 每一步之后，运行该步骤指定的验证（测试、类型检查、lint、构建检查、冒烟测试）。
5. 如果某一步之后验证失败，先停下来。判断失败属于哪种：
   - 重构引入的回归（修复或撤销该步骤）
   - 测试原本测的是实现细节（更新测试，但要记录改了什么、为什么）
   - 与重构无关的既有失败（记录并继续）
6. 不要改变可观察行为。如果某一步需要行为变更，先停下来提问。
7. 不要把重构与功能开发、缺陷修复或无关清理混在一起。
8. 遵循现有代码风格和约定。
9. 未经我明确许可，不要提交或推送。
10. 如果执行过程中发现计划需要调整，先停下来解释，再继续。

可观察行为包括：
- 公共函数的返回值、副作用和错误行为
- API 响应（状态码、body 结构、header）
- 数据库写入（schema、数据形状、查询行为）
- UI 渲染和用户可见文案
- 被下游系统依赖的日志输出
- 事件发出和消息 payload

结束时请提供：

1. 已完成的变换步骤
2. 变更文件
3. 更新过的测试及原因（实现细节测试 vs. 回归问题）
4. 每一步之后的验证结果
5. 跳过或调整的步骤，以及原因说明
```

## Step 4：验证重构

执行完成后，或在 Step 3 修复过问题之后，使用下面提示词：

```text
根据原始目标和已记录的验证基线，验证这次重构。

重构目标：

<要重构什么，以及为什么>

基线：

<来自 Step 0 的验证命令、基线结果和已知既有失败>

执行摘要：

<来自 Step 3 的已完成变换步骤、已更新测试和验证结果>

你的目标：

1. 重新运行 Step 0 和 Step 2 中建立的完整验证集（如有），然后与记录的基线对比。
   - 通过项不变且没有新增失败：很好。
   - 出现新增失败：调查它是回归、实现细节测试变化，还是无关问题。
   - 现在跳过了基线或计划中的检查：说明原因。
   - 移除了或弱化了测试：逐项说明理由。
2. 验证所有公共接口保持稳定：
   - 函数签名、返回类型、错误类型
   - API 契约
   - 数据库 schema 和查询行为
   - 导出项和模块边界
3. 审查 diff，确认没有无意的行为变化：
   - 返回值或错误信息发生变化
   - 带副作用的操作顺序被调整
   - 改变了会影响时序或并发的控制流
   - 无意删改日志、指标或事件
4. 验证这次重构是否真的达成目标。代码结构是否确实得到改善？
5. 检查是否存在未完成的变换：旧模式残留、不一致命名、因重构本身引入的孤儿文件或死代码。
6. 如果验证显示这次重构比最初分类更广或更危险，明确指出，并建议在继续前是否需要额外规划或缩小范围。

未经我明确许可，不要提交或推送。

请提供：
- 与基线对比的验证结果（逐命令对照 before vs. after）
- 公共接口稳定性检查（stable / changed，如果 changed 需说明理由）
- 发现的非预期行为变化（none / list）
- 是否达成目标（yes / partially / no，并说明原因）
- 是否还需要清理（none / list）
```

在没有未解决回归、不稳定接口或未完成变换之前，反复执行 Step 3 与 Step 4。

推荐停止条件：

```text
如果 Step 4 发现新的回归、非预期行为变化、接口不稳定或未完成变换，回到 Step 3 修复，再重新执行 Step 4。当记录的验证集相对于基线可以接受，且没有未解决的重构风险时，停止循环。
```

## Step 5：最终报告

对于 Architectural 重构，或在 Step 4 清理完成后需要交接总结时，使用下面提示词：

```text
输出最终重构报告。

重构目标：

<要重构什么，以及为什么>

报告应包含：

1. 重构摘要（改了什么，为什么）
2. Before and after：结构对比（模块、依赖、接口）
3. 变更文件
4. 更新、新增或移除的测试（每个被移除测试都要说明理由）
5. 验证结果（基线 vs. 最终结果，覆盖测试、类型检查、lint、构建和冒烟）
6. 公共接口：确认保持稳定，或说明变化及理由
7. 行为变化：确认没有，或说明变化及理由
8. 未完成项或后续工作
9. 确认未执行提交或推送

未经我明确许可，不要提交或推送。
```

## 实务说明

- 另外也要遵循 [shared/workflow-conventions.md](shared/workflow-conventions.md) 中的共享约定。
- 首要规则：不要改变行为。如果你需要改变行为，那就不是重构，而是功能改动或缺陷修复，应使用相应工作流。
- 以增量方式工作。一次触及整个系统的大重构很难验证，也很难回滚。每一步都应让代码保持可工作状态。
- 区分实现细节测试和行为测试。重构理所当然会打破一些断言内部结构的测试（例如 mock 调用次数、私有方法名、具体 SQL 字符串），这些测试应当更新。断言可观察行为的测试则**不应该**被打破；如果它们失败了，就说明重构改变了行为。
- 不要把重构和其他工作混在一起。一个既重构又顺便修 bug 或加功能的 PR，会更难评审，也更难回滚。
- 如果受影响代码没有测试，认真考虑先补行为测试。对无测试代码做重构风险很高，因为没有安全网来抓回归。
- 如果重构暴露出设计缺陷或 bug，把它记录下来，单独处理。不要把它顺手修在这次重构里。
