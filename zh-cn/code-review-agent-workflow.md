# 代码评审代理工作流

本文定义了一套可复用的代理工作流，用于评审代码变更。输入可以是 GitHub Pull Request 链接、分支名，或工作区中尚未提交的变更。

该工作流面向在仓库中工作的 AI 编码代理，覆盖正确性、安全性、性能、测试覆盖、可维护性和兼容性。它输出结构化发现项，除非被明确要求，否则不会直接修复问题。

## 预检仓库检查

在进入分诊门之前，使用 [代码评审预检](shared/repository-preflight.md#code-review-preflight)。

## 全局规则

在每一个评审提示词中使用 [仅评审规则](shared/safety-rules.md#review-only-rule)。

## 分诊门

开始之前，先对评审任务分类，以决定需要多深入：

```text
将这次代码评审归类到以下类别之一：

1. Quick：小且低风险的改动。文件少、范围窄，没有数据模型或 API 影响。例子：文案修复、配置改动、测试新增、样式清理、依赖升级。

2. Standard：中等规模的改动，包含有意义的逻辑。涉及多个文件，或者对用户、API 或数据有影响。例子：新增接口、业务逻辑变更、权限更新、UI 功能。

3. Deep：大改动或高风险改动。存在跨领域影响、schema 迁移、安全敏感代码、性能关键路径或向后兼容约束。例子：认证改动、支付逻辑、数据迁移、新集成、基础设施改动。

根据分类：

- Quick：跳到 Step 1（Review）。重点检查正确性和显而易见的问题。如果评审确认这次变更确实低风险，一次评审通过通常就够。**升级检查：**如果评审过程中发现超出预期的安全、兼容性、迁移、性能或测试风险，请暂停并重新分类为 Standard 或 Deep，再继续。
- Standard：执行 Step 0 到 Step 2。覆盖所有相关评审维度，并报告已运行或跳过的验证命令。
- Deep：完整执行 Step 0 到 Step 3。对安全、性能、兼容性和边界场景施加更高审查强度。

待评审代码：

<GitHub PR 链接、分支名，或“当前工作区变更”>

请对这次评审分类，并建议应遵循哪些步骤。
```

## Step 0：理解上下文

评审前先建立上下文，使用下面提示词：

```text
在评审代码之前，先理解这次变更的上下文。

待评审代码：

<GitHub PR 链接、分支名，或“当前工作区变更”>

你的目标：

1. 搞清楚这次变更想实现什么。
   - 如果提供的是 GitHub PR 链接，阅读 PR 标题、描述、评论、关联 issue 和标签。
   - 如果提供的是分支名，阅读提交历史以及相对正确基线分支的 diff。
   - 如果评审的是工作区变更，阅读所有在本次评审范围内的 staged 和 unstaged diff。
2. 识别范围：哪些文件、模块或子系统受到影响。
3. 判断变更类型：功能、缺陷修复、重构、依赖更新、配置、测试、文档，或混合类型。
4. 记录影响面：哪些现有行为、调用方、使用方或契约可能受影响。
5. 检查是否存在相关设计文档、issue 或规范来定义期望行为。
6. 判断比较基线是否正确且足够明确，以支持可靠评审。
   - 如果评审 PR，记录 PR 元数据中的基线分支。
   - 如果评审分支，记录所选基线分支以及为什么它是正确比较目标。
   - 如果评审工作区变更，把评审目标与无关本地编辑分离开。
7. 如果评审目标或比较基线仍有歧义，先停下来提问，再继续。

先不要开始评审。
未经我明确许可，不要改代码、提交或推送。

请提供：
- 变更摘要（改了什么，为什么）
- 变更类型
- 受影响的文件和模块
- 影响面
- 评审目标和比较基线
- 相关上下文（关联 issue、设计文档、PR 描述），如果有
```

## Step 1：评审代码

在上下文明确之后（或直接用于 Quick 评审），使用下面提示词：

```text
评审以下代码变更。

待评审代码：

<GitHub PR 链接、分支名，或“当前工作区变更”>

上下文：

<来自 Step 0 的变更摘要、变更类型和影响面；若跳过 Step 0，则填写简短说明>

从下面这些维度评审该变更。某个维度显然不适用时，可以跳过。

### Correctness
- 代码是否真的实现了它声称实现的事情？
- 是否存在逻辑错误、off-by-one、错误条件或遗漏的情况？
- 是否处理了边界场景（null、空输入、边界值、并发访问）？
- 错误处理是否覆盖了失败模式，而不是默默吞掉错误？

### Security
- 这次改动是否引入注入风险（SQL、命令、XSS、模板）？
- 系统边界上的输入是否经过校验和清洗？
- 密钥、令牌或凭证是否被安全处理（没有硬编码、没有写日志）？
- 权限和授权检查是否正确且完整？
- 敏感数据（PII、凭证、令牌）在传输、存储和日志中是否受到保护？

### Performance
- 这次改动是否引入了不必要的数据库查询、API 调用或循环？
- 是否存在 N+1 查询模式、无界迭代或缺少分页？
- 在高负载或大数据集下，这次改动会不会让性能恶化？
- 高成本操作是否按需进行了缓存、批处理或延后执行？

### Test Coverage
- 新增或变更的行为是否有测试？
- 测试是否覆盖 happy path、边界场景和错误场景？
- 如果回滚这次改动，测试是否会失败（具备回归价值）？
- 测试断言是否足够具体，能抓住真实回归？
- 是否存在应该更新但没有更新的现有测试？

### Maintainability
- 代码是否遵循代码库现有模式和约定？
- 命名是否清晰，并与领域语言一致？
- 变更范围是否合理，还是混入了无关关注点？
- 是否存在不必要的复杂性、重复或抽象？

### Compatibility
- 这次改动是否破坏现有 API 契约、接口或数据格式？
- 在需要时，是否保留了向后兼容？
- 数据库迁移是否安全（可回滚、大表上避免锁表、先加法后破坏）？
- 是否需要特性开关或灰度门？

每个发现都使用下面格式：

### [Severity] Finding title

- Location: file path and line range
- Problem: what is wrong
- Why it matters: impact if not addressed
- Suggested fix: concrete recommendation
- Severity: Critical / Major / Minor / Nit

严重级别定义：
- Critical：会导致数据丢失、安全漏洞、崩溃或功能损坏。合并前必须修复。
- Major：行为错误、缺少边界场景处理或存在重大风险。合并前应该修复。
- Minor：代码质量、可维护性或轻微正确性问题。建议修复，但不阻塞。
- Nit：风格、命名或偏好问题。可选。

在最终给出评审结论前：
- 如果变更看起来比当前评审类别更高风险，请升级到 Standard 或 Deep，并先完成缺失步骤，再给最终判断。
- 在可行时运行相关验证命令（测试、lint、类型检查、定向评审辅助工具），并报告结果。
- 如果某些验证无法运行，要明确说明原因，并相应降低最终结论的置信度。

未经我明确许可，不要改代码、提交或推送。

结束时请提供：
- 各严重级别发现总数
- 总体结论：Approve / Approve with minor fixes / Request changes
- 如果有问题，用一句话总结最重要的发现
- 运行过的验证命令及结果，或跳过的命令及原因
```

## Step 2：深入评审测试覆盖

当是 Standard 或 Deep 评审且需要更细查测试覆盖时，使用下面提示词：

```text
评审以下代码变更的测试覆盖情况。

待评审代码：

<GitHub PR 链接、分支名，或“当前工作区变更”>

你的目标：

1. 列出这次代码变更引入或修改的每一个行为。
2. 针对每个行为，检查是否存在测试，且该测试：
   - 直接覆盖该行为
   - 在该行为被移除或损坏时会失败
   - 覆盖了相关边界场景
3. 识别缺失的测试场景。要具体：描述输入、期望行为和为什么重要。
4. 评审被修改的现有测试：
   - 这些修改是否必要且正确？
   - 修改是否削弱了测试（例如放宽断言、删除用例）？
5. 检查测试质量问题：
   - 无论实现如何都会通过的测试（恒真测试）
   - 脆弱测试（依赖时间、顺序或外部状态）
   - 重复覆盖但不增加价值的测试
6. 在可行时运行相关现有测试命令，确认测试套件或定向测试能按预期执行。
7. 如果当前环境无法运行测试，请明确说明原因。

未经我明确许可，不要改代码、提交或推送。

请提供：
- 覆盖矩阵：behavior vs. test（covered / partially covered / missing）
- 缺失的测试场景以及建议的测试描述
- 测试质量问题（如果有）
- 运行过的验证命令及结果，或跳过的命令及原因
```

## Step 3：深度专项检查（安全、性能或兼容性）

当是 Deep 评审且需要对某个维度做额外深入检查时，使用下面提示词：

```text
对以下代码变更做一次聚焦型深度评审。

待评审代码：

<GitHub PR 链接、分支名，或“当前工作区变更”>

关注领域：

<"security" | "performance" | "compatibility" — 选择一个或指定多个>

### 如果是 security：
1. 追踪每一个外部输入从进入系统到最终使用的位置，识别所有注入点。
2. 检查受变更影响的每个入口点上的认证和授权逻辑。
3. 检查错误信息和日志是否泄露信息。
4. 检查是否存在时序攻击、TOCTOU 问题或不安全默认值。
5. 验证密钥管理是否遵循项目约定。
6. 检查新增或升级的依赖是否有已知漏洞。

### 如果是 performance：
1. 追踪这次变更影响到的热点路径，识别最昂贵的操作。
2. 估算在现实负载下的性能影响（典型情况和最坏情况）。
3. 检查是否缺少索引、存在未优化查询或不必要的数据抓取。
4. 审查内存分配模式（循环里创建大对象、无界缓存、引用泄漏）。
5. 判断在合并前是否值得先做性能剖析或基准测试。

### 如果是 compatibility：
1. 列出所有受影响的公共 API、接口、数据格式或契约。
2. 对每一项判断这次改动是加法式、破坏式还是含糊不清。
3. 检查迁移安全性：是否能无停机部署？能否回滚？
4. 找出可能被影响但没有在这次变更中一起更新的调用方或使用方。
5. 检查是否需要版本管理、废弃声明或特性开关。

每个问题都使用标准发现格式。

另外：
- 在可行时运行与你选择的 focus area 相关的验证命令（例如定向测试、静态分析、安全扫描、基准测试、迁移检查或契约检查）。
- 如果某个验证步骤无法执行，要明确说明原因。

未经我明确许可，不要改代码、提交或推送。

请提供：
- 按严重级别排列的发现项
- 对该 focus area 的风险总结
- 缓解建议（如果有）
- 运行过的验证命令及结果，或跳过的命令及原因
```

## Optional：修复发现项

只有在被明确要求修复评审发现项时，才使用下面提示词：

```text
在当前工作区中修复下面列出的评审发现项。

待修复发现项：

<粘贴评审中的发现项，或填写“all Critical and Major findings”>

重要说明：

1. 编辑前仔细阅读每个发现项。
2. 只修复列出的发现项。不要重构、清理或改进周边代码。
3. 遵循代码库现有模式。
4. 任何正确性修复都要补充或更新测试。
5. 修复后运行相关测试、lint 和类型检查。
6. 未经我明确许可，不要提交或推送。
7. 如果某个发现项有歧义，或者修复会带来副作用，先停下来提问。
8. 修复完成后，要回到评审模式验证目标发现项是否已解决，再认为工作结束。

结束时请提供：

1. 已修复的发现项（附文件和行号）
2. 未修复的发现项及原因
3. 新增或更新的测试
4. 运行过的命令及结果
```

## 修复后复评

在执行 Optional：Fix Findings 之后，使用下面提示词：

```text
在请求修复的评审发现项处理完之后，重新评审当前工作区。

原始评审目标：

<GitHub PR 链接、分支名，或“当前工作区变更”>

本应被修复的发现项：

<粘贴已修复的发现项>

你的目标：

1. 验证每个目标发现项都确实被解决。
2. 检查这些修复是否引入新的正确性、安全性、性能、兼容性或测试问题。
3. 重新运行相关验证命令。
4. 确认修复过程中没有编辑无关文件。
5. 使用标准发现格式报告仍未解决的发现项或新引入的发现项。

未经我明确许可，不要改代码、提交或推送。

结束时请提供：
- 已确认修复的发现项
- 仍然开放的发现项
- 修复引入的新发现项（如果有）
- 运行过的验证命令及结果，或跳过的命令及原因
- 总体结论：Ready / More fixes required
```

重复执行 Optional：Fix Findings 与 Post-Fix Re-review，直到没有未解决的 Critical 或 Major 问题。

## 实务说明

- 另外也要遵循 [shared/workflow-conventions.md](shared/workflow-conventions.md) 中的共享约定。
- 默认保持只读评审。除非用户明确要求你修复发现项，否则不要改代码。
- 使用分诊门，避免对琐碎改动做过度评审，也避免对高风险改动审得不够深。
- 发现项要具体。“这里还可以更好”没有可执行性；“第 42 行 `name` 参数可以导致 SQL 注入”才是可操作的。
- 严重级别要有分寸。要区分“会炸生产”和“可以更整洁一点”。对每个 nit 都大喊大叫的评审不会被信任。
- 评审 GitHub PR 时，要读完整个 diff，而不是只看文件列表。变更周围的上下文行往往能暴露被忽略的边界场景。
- 不要拿一个“猜测的基线分支”去评审分支。先明确比较基线，否则就提问。
- 如果变更大到无法在一轮里有效评审，就明确指出，并建议拆分。因范围过大而漏掉问题的评审，比不评审更糟。
- 如果你修复了发现项，在宣布评审完成前，先重新评审修复结果。
- 如果评审暴露出设计层面问题（不是实现错了，而是方案本身就不对），要尽早指出，而不是列一堆一旦设计改变就全都失效的实现层问题。
