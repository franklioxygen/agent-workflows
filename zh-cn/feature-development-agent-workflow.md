# 功能开发代理工作流

本文定义了一套可复用的代理工作流，用于将中大型功能从需求接收推进到设计、评审、实现、验证和最终交接。

该工作流面向在仓库中工作的 AI 编码代理。它适用于存在明显歧义、跨文件变更、用户可见行为变化、数据模型影响、API 影响、权限、迁移、发布风险或非平凡测试需求的功能。对于小型缺陷修复或范围很窄的改动，应使用更轻量的工作流，而不是强行把每一步都走完。

它强调需求清晰、谨慎设计、结构化评审、范围受控的实现、测试覆盖、最终验证，以及避免未经授权的提交或推送。

## 预检仓库检查

在进入分诊门之前，使用 [标准编码预检](shared/repository-preflight.md#standard-coding-preflight)。

## 分诊门

开始该工作流前，先对变更分类，以选择合适的流程深度：

```text
将这次变更归类到以下类别之一：

1. Small：单文件或范围很窄的改动。歧义低。没有数据模型、API、权限或迁移影响。例子：缺陷修复、文案修改、配置更新、增加一个直观的工具函数。

2. Medium：跨文件改动，并存在一定歧义。可能触及数据模型、API 或用户可见行为，但整体范围已被很好理解。例子：新增一个 API 端点、扩展现有功能、增加一条新的校验规则。

3. Large：歧义明显、存在跨领域问题，或风险较高。涉及数据迁移、权限变化、新集成、向后兼容约束或发布风险。例子：带 schema 变更的新功能、认证系统改动、计费逻辑、多服务协同。

根据分类：

- Small：使用下面的 Small-Change Implementation Prompt，替代 Step 0 到 Step 6。把变更说明和验收标准直接写在输入里。之后运行验证命令。
- Medium：从 Step 0（澄清需求）开始。设计文档（Step 1）可以省略所有标为“if any”的不适用章节。如果没有发现 Critical 或 Major 问题，一轮设计评审（Step 2）即可。Step 3 和 Step 4 可以在同一会话中完成，但在先给出实现计划之前，不要编辑代码。
- Large：完整执行 Step 0 到 Step 6。

Change description:

<brief description of the change>

请对这次变更分类，并建议应遵循哪些步骤。
```

## 全局规则

在每个实现或评审提示词中使用 [标准编码规则](shared/safety-rules.md#standard-coding-rule)。

## Small-Change Implementation Prompt

对于分诊门中归类为 `Small` 的改动，使用下面提示词替代 Step 0 到 Step 6：

```text
实现下面这个小型、低风险改动，不需要创建完整设计文档。

Change description:

<brief description of the change>

验收标准：

<验收标准>

重要说明：

1. 检查现有代码库并遵循已有模式。
2. 让改动严格限定在描述的任务范围内。
3. 如果你发现存在歧义、跨文件风险、数据模型影响、API 影响、权限影响、迁移影响或发布风险，先停下来，建议切换到完整功能工作流。
4. 如果改动影响行为，就添加或更新测试。
5. 运行相关验证命令。
6. 未经我明确许可，不要提交或推送。
7. 不要覆盖或回退与当前任务无关的工作区变更。

结束时请提供：

1. 变更文件
2. 已实现的行为
3. 新增或更新的测试
4. 运行过的命令及结果
5. 已知风险或后续事项
```

## Step 0：澄清需求

在创建设计文档前，使用下面提示词：

```text
审查下面的功能需求，并对现有代码库做足够检查，以识别歧义、缺失上下文和实现风险。

需求：

<功能概述>
<执行逻辑 1、2、3……>
<验收标准>

你的目标：

1. 判断需求是否已经足够清晰，可以安全进入设计。
2. 识别会影响以下方面的需求：数据模型、API 契约、用户可见行为、安全、隐私、权限、迁移、计费、日志、向后兼容或发布风险。
3. 将结果拆分为：
   - Confirmed requirements
   - Inferred assumptions
   - Open questions
4. 当答案会改变产品行为、API 行为、schema 形状、权限逻辑、迁移策略或兼容性时，在定稿设计前先提澄清问题。
5. 除非需求已经足够清晰且剩余假设风险很低，否则先不要创建设计文档。

未经我明确许可，不要提交或推送。

如果需求已经足够清晰，请明确说明，并给出 confirmed requirements、inferred assumptions 以及所有低风险 open questions。

如果需求还不够清晰，只问那些能解除设计阻塞的问题。
```

## Step 1：创建设计文档

当需求已经足够清晰后，使用下面提示词：

```text
根据下面的需求创建详细的功能设计文档。

将文档写成 Markdown。若仓库里已有 `reports/` 目录，则放入其中；否则使用仓库原有的设计文档位置。如果没有明显约定，就创建一个 `reports/` 目录。
使用清晰的文件名，包含功能名和当前日期，例如：

`reports/<feature-name>-design-YYYY-MM-DD.md`

文档应包含：

1. 功能概览
2. 已确认需求
3. 目标与非目标
4. 当前行为 / 现有系统上下文
5. 提议设计
6. 执行逻辑
   - Step-by-step flow
   - Edge cases
   - Error handling
   - Data validation
7. API / 接口变更（如果有）
8. 数据库 / Schema 变更（如果有）
9. UI / UX 变更（如果有）
10. 安全 / 隐私 / 权限考虑（如果有）
11. 向后兼容考虑（如果有）
12. 测试计划
13. 发布 / 迁移计划（如果有）
14. 假设与依赖
15. 开放问题
16. 实现检查清单
17. 完成定义

省略所有不适用的“如果有”章节。不要用占位文字或填充内容凑格式。

需求：

<功能概述>
<已确认需求>
<执行逻辑 1、2、3……>
<验收标准>

如果某段设计逻辑存在更优方案，请直接修正，并解释原因。

如果任何点不清晰或有风险，把它列到 “Open Questions（开放问题）” 下，并在做假设前先问我。

所有假设都要明确记录。不要把猜测当成需求。

设计文档必须足够详细，让另一位工程师无需额外上下文就能直接实现。

未经我明确许可，不要提交或推送。

现在开始。
```

### 建议的完成定义

把下面这节加入每个设计文档：

```markdown
## 完成定义

- 功能行为与已确认需求一致。
- 验收标准已被覆盖。
- 边界场景已处理。
- 已处理安全、隐私和权限方面的考虑。
- 已新增或更新测试。
- 相关验证命令全部通过。
- 没有未解决的开放问题影响已实现行为。
- 没有修改无关文件。
- 设计检查清单已全部标记完成。
- 已记录已知限制。
```

## Step 2：评审设计文档

设计文档创建完成后，新开一个会话并使用下面提示词：

```text
存在一个功能设计文档，路径为：

<path-to-design-document>

请仔细评审这份设计。

你的目标：

1. 检查设计是否完整，是否已经可以进入实现。
2. 识别缺失需求、不清晰逻辑、边界场景、兼容性风险、安全风险、隐私风险、权限风险、迁移风险或测试缺口。
3. 验证提议设计是否适配现有代码库，且不会引入不必要的风险。
4. 按严重级别分类每个发现：
   - Critical：实现前必须修复
   - Major：实现前应该修复
   - Minor：改进或清理
   - Question：需要产品或工程澄清
5. 对于事实性修正和明显缺失的章节，直接更新文档。
6. 对于产品决策、API 行为、schema 变更、权限行为、迁移策略或取舍决策，不要静默修改；把它们写成 “Review Findings（评审发现）” 或 “Open Questions（开放问题）”。
7. 如果你做了修改，新增或更新 “Review Findings（评审发现）” 章节，说明改了什么、为什么改。

每个发现都使用以下格式：

### [Severity] Finding title

- Location:
- Problem:
- Why it matters:
- Recommended fix:
- Status: Open / Fixed / Needs decision

先不要实现这个功能。
未经我明确许可，不要提交或推送。

如果文档已经可以直接实现，请明确说明，不要做不必要修改。
```

在没有有意义问题之前，可在新会话中重复此步骤。

推荐停止条件：

```text
当两次连续、彼此独立的评审都没有发现 Critical 或 Major 问题时，停止重复设计评审。
```

## Step 3：创建实现计划

当设计文档稳定后，在编辑代码前使用下面提示词：

```text
已有一个定稿的功能设计文档，路径为：

<path-to-design-document>

阅读该设计文档并检查现有代码库。编辑代码前，先给出简洁的实现计划。

计划应包含：

1. 可能要改动的文件
2. 需要遵循的现有模式
3. 要新增或更新的测试
4. 要运行的验证命令
5. 会阻塞实现的未解决 Open Questions
6. 实现过程中需要重点注意的安全、隐私、权限、迁移或兼容性风险

不要实现任何依赖未解决 Open Questions 的部分。
未经我明确许可，不要提交或推送。

如果计划暴露出歧义或风险，先停下来请求澄清，再编辑代码。
如果计划没有暴露出阻塞性的歧义或风险，先展示计划，然后等待明确批准，再继续到 Step 4。
```

## Step 4：实现功能

当实现计划明确后，使用下面提示词：

```text
已有一个定稿的功能设计文档，路径为：

<path-to-design-document>

本提示词假定定稿设计文档已经存在。对于分诊门中的 `Small` 改动，请使用 Small-Change Implementation Prompt。

根据设计文档和实现计划开始实现该功能。

重要说明：

1. 编辑代码前先仔细阅读设计文档。
2. 检查现有代码库并遵循已有模式。
3. 如果仓库流程允许，则创建新分支；否则解释为什么留在当前分支。
4. 按步骤实现功能。
5. 不要实现任何依赖未解决 Open Questions 的部分。
6. 让改动严格限定在当前功能范围内。
7. 未经我明确许可，不要提交或推送。
8. 根据测试计划新增或更新测试。
9. 在可用时运行相关测试、lint 和类型检查。
10. 如果实现过程中发现设计问题，先停下来记录，再决定方案。
11. 只在到达实质性里程碑或实现结束时更新 implementation checklist。避免噪音式的“只改 checklist”提交。

质量标准：

这个功能必须达到可上生产的程度。务必仔细处理兼容性、边界场景、错误处理、安全、隐私、权限、迁移和测试覆盖。

结束时请提供：

1. 变更文件
2. 已实现的行为
3. 新增或更新的测试
4. 运行过的命令及结果
5. 已知风险或后续事项

现在开始。
```

## Step 5：评审工作区变更

功能实现后，新开一个会话并使用下面提示词：

```text
根据路径为下述位置的设计文档，评审当前工作区变更：

<path-to-design-document>

你的目标：

1. 将实现与设计进行对照。
2. 识别缺失功能、错误行为、边界场景、回归、安全风险、隐私风险、权限缺口、迁移风险或测试缺口。
3. 仔细审查变更文件。
4. 按严重级别分类每个发现：
   - Critical：合并前必须修复
   - Major：合并前应该修复
   - Minor：改进或清理
   - Question：需要产品或工程澄清
5. 如果发现问题，在设计文档中以 “Implementation Review Findings（实现评审发现）” 记录。
6. 如果问题安全且明显在当前范围内，可以直接修复。
7. 修复后运行相关验证命令。
8. 未经我明确许可，不要提交或推送。

每个发现都使用以下格式：

### [Severity] Finding title

- Location:
- Problem:
- Why it matters:
- Recommended fix:
- Status: Open / Fixed / Needs decision

如果没有问题，请明确说明实现与设计一致，并列出执行过的验证。
```

在没有问题前，可在新会话中重复该评审。

推荐停止条件：

```text
当两次连续、彼此独立的实现评审都没有发现 Critical 或 Major 问题时，停止重复实现评审。
```

## Step 6：最终验证 / 交接

实现评审完成后，使用下面提示词：

```text
根据路径为下述位置的设计文档，对已实现功能做最终验证：

<path-to-design-document>

输出一份最终验证报告，包含：

1. 需求覆盖
2. 验收标准覆盖
3. 变更文件
4. 新增或更新的测试
5. 运行过的命令及结果
6. 已知风险
7. 手工 QA 清单
8. 有意未实现的项
9. 确认未执行提交或推送

另外还要验证：

1. 没有未解决的 Open Questions 影响已实现行为。
2. 安全、隐私和权限考虑已处理，或被明确标记为不适用。
3. 与该仓库相关的验证命令已运行，或对跳过原因有清晰说明。

未经我明确许可，不要提交或推送。
```

## 实务说明

- 另外也要遵循 [shared/workflow-conventions.md](shared/workflow-conventions.md) 中的共享约定。
- 在整个流程中，把设计文档视为唯一事实来源。
- 这个完整工作流适用于中大型功能。对于小型、低风险改动，更应该使用清晰需求、范围受控实现和验证组成的轻量流程。
- 除非答案能从现有代码库里直接、明显地推出，否则把 open questions 当作阻塞项。
- 不要实现任何依赖未解决 Open Questions 的行为。
- 与其给出模糊评论，不如给出具体发现项。
- 对设计评审和实现评审都使用结构化发现格式。
- 对于产品决策、API 行为、schema 变更、权限行为、迁移策略或取舍决策，不要静默调整设计。
- 避免只改 checklist 的噪音式编辑。应在完成有意义的里程碑或实现结束时再更新 checklist。
- 提示词里不要制造情绪压力。明确的质量标准和具体的验证步骤，效果更好。
