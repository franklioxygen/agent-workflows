# 技术债 / 清理代理工作流

本文定义了一套可复用的代理工作流，用于处理技术债、清理任务、依赖升级、死代码移除、测试覆盖缺口和 TODO 项。输入可以是明确的债务项、某一类债务的盘点目标，或一个宽泛的清理目标。

该工作流面向在仓库中工作的 AI 编码代理。它与重构在范围和目的上不同：

- 重构：重组特定代码，以在保持行为不变的前提下改善设计。
- 技术债清理：在整个代码库范围内盘点、排序并解决累积问题，例如死代码、过期依赖、缺失测试、陈旧 TODO、不一致模式和废弃 API。

技术债工作非常容易发生范围蔓延。整个流程的重点都是控制范围、排序优先级，并以增量方式完成。

## 预检仓库检查

在进入分诊门之前，使用 [标准编码预检](shared/repository-preflight.md#standard-coding-preflight)。

继续之前，增加一项工作流专属预检：

- 检查现有的技术债跟踪信息（TODO 注释、issue 标签、债务 backlog、废弃通知），避免重复做已知工作。

## 全局规则

在每个提示词中使用 [清理规则](shared/safety-rules.md#cleanup-rule)。

## 分诊门

开始之前，对清理任务分类，以选择合适的流程深度：

```text
将这次技术债 / 清理任务归类到以下类别之一：

1. Targeted：一个明确、定义清晰的清理项。范围清楚，风险低。例子：删除单个死文件、移除未使用 import、处理一个明确 TODO、更新一个废弃 API 调用、移除一个已经完全放开的特性开关。

2. Scoped：跨多个文件或某个模块的、有边界的清理工作。范围已定义，但需要先做盘点。例子：移除整个代码库中某个废弃工具的所有使用、为一个模块补缺失测试、合并重复校验逻辑、在全代码库升级某个特定依赖。

3. Survey：范围尚未定义。目标是在某个区域或整个代码库里评估技术债现状，再决定清理什么。例子：“我们有哪些技术债？”“盘点测试覆盖缺口”“找出所有废弃 API 用法”“代码里有哪些 TODO？”

根据分类：

- Targeted：从 Step 1（建立基线）开始。如果基线确认该任务确实很小且低风险，则进入 Step 3（执行）。除非风险在过程中变大，一轮执行/验证循环通常就够。
- Scoped：从 Step 1（建立基线）开始，然后进入 Step 2（规划）。如果范围仍需调查或排序，就先执行 Step 0（评估）。
- Survey：从 Step 0（评估）开始。以盘点报告结束。除非用户要求，否则不要改代码。

清理任务：

<技术债或清理目标的描述>

请对这个任务分类，并建议应遵循哪些步骤。
```

## Step 0：评估技术债

对于 Survey 任务，或者清理范围仍不清楚时，使用下面提示词：

```text
评估下面描述区域中的技术债。不要改任何代码。

清理目标：

<要评估的区域或类别描述>

你的目标：

1. 针对指定债务类别对代码库做盘点。常见类别包括：
   - Dead code（未使用函数、不可达分支、孤儿文件、被注释掉的代码）
   - Deprecated APIs or patterns（内部或外部）
   - TODO / FIXME / HACK 注释
   - 缺失或不足的测试覆盖
   - 过期依赖（主版本落后、已知漏洞、无人维护）
   - 不一致模式（同一件事在不同地方有不同做法）
   - 陈旧配置（未使用 env 变量、死掉的特性开关、过时的构建目标）
   - 文档漂移（文档与代码不再一致）
2. 对每个找到的项，记录：
   - Location（文件和行号）
   - Category
   - Description（这是什么债）
   - Risk if left unaddressed（如果不处理，会带来什么问题或成本）
   - Effort estimate（trivial / small / medium / large）
   - Dependencies（修它之前是否必须先修别的东西）
3. 按类别对这些项分组。
4. 根据以下因素推荐优先级顺序：
   - 风险（安全、正确性、数据完整性优先）
   - Effort-to-value ratio（先做 quick win）
   - Dependencies（先清理能解锁其他工作的项）
   - Blast radius（先做隔离的改动，再做跨领域改动）

不要改任何代码。
未经我明确许可，不要提交或推送。

请提供：
- 按类别分组的债务项清单
- 优先级建议
- 不应立即清理的项（承重 workaround、故意保留的设计、需要设计决策的改动）
- 建议的下一步：先做哪些项，以及后续应走 Targeted 还是 Scoped 路径
```

## 盘点报告（Survey Report）

对不改代码的 Survey 任务，在 Step 0 之后使用下面提示词：

```text
输出技术债盘点报告。

清理目标：

<原始评估目标>

评估摘要：

<来自 Step 0 的债务项清单、分类结果、优先级和排除项>

报告应包含：

1. 按类别的清单摘要
2. 最高优先级的债务项，以及它们为什么优先
3. 暂时不应该清理的项，以及原因
4. 债务项之间的依赖或执行顺序约束
5. 建议的下一步：
   - 先处理哪个项或哪一组项
   - 下一轮应走 Targeted 还是 Scoped 路径
6. 确认未改动任何代码
7. 确认未执行提交或推送

不要改代码。
未经我明确许可，不要提交或推送。
```

## Step 1：建立基线

对于任何 Targeted 或 Scoped 清理，在编辑代码前使用下面提示词：

```text
开始清理前建立验证基线。

清理目标：

<清理任务描述>

你的目标：

1. 识别与该清理相关的验证命令：
   - Tests
   - Type checks
   - Linters or format checks
   - Build/package commands
   - Usage checks（grep、搜索、import/export 校验、依赖图检查）
   - Manual smoke tests or targeted runtime checks
2. 现在就运行相关验证集合，并记录结果。
3. 将结果拆分为：
   - Passing baseline checks
   - Pre-existing failures
   - Missing coverage or validation gaps
4. 判断当前安全网是否足够开始清理。
5. 如果基线暴露出范围更大、覆盖偏弱或存在行为风险，建议是否应：
   - 缩小范围
   - 先补测试或额外检查
   - 将任务从 Targeted 重新归类为 Scoped 或 Survey
6. 记录适用于该清理类别的特殊检查：
   - 死代码移除：使用情况验证和动态引用检查
   - 依赖升级：changelog 审阅、迁移指南审阅、audit/security 检查
   - TODO/FIXME 清理：如果当前范围内无法解决，需要补 issue 或跟踪项

现在先不要编辑代码。
未经我明确许可，不要提交或推送。

请提供：
- 清理过程中要用的验证命令
- 已通过的基线检查
- 需要关注的既有失败
- 覆盖缺口与风险
- 建议的下一步
```

后续步骤默认这份基线已经记录，并会在每一轮执行后拿来对比。

## Step 2：规划清理

对于范围已明确的 Scoped 任务，使用下面提示词：

```text
为下面这次技术债清理创建计划。

清理目标：

<清理任务描述，以及来自 Step 0 的盘点结果（如果有）>

基线：

<来自 Step 1 的验证命令、基线结果和覆盖缺口>

计划应包含：

1. 按执行顺序排列的清理项。对每一项说明：
   - 要改什么
   - 涉及哪些文件
   - 为什么纳入此次清理（不是“因为它旧了”，而是它造成了什么成本）
   - 风险：这项改动是否可能破坏某些东西？现有代码依赖了什么假设？
   - 验证：如何确认这项清理是安全的（测试、类型检查、grep usages）
2. 明确排除在这次清理之外的项，以及原因（超出范围、需要设计决策、没有更多测试就太危险、属于故意保留的 workaround）。
3. 依赖顺序：哪些项必须先做，才能做后面的项。
4. 已记录的基线验证：Step 1 中哪些检查将用于发现回归。
5. 范围边界：一句话清楚说明本次清理**会做什么**、**不会做什么**。如果很容易发生 scope creep，要点名需要刻意避免的“顺手修一下”诱惑。

现在先不要编辑代码。
未经我明确许可，不要提交或推送。

如果计划显示某些项比预期更危险，或需要设计决策，请明确指出，并建议单独处理。
```

## Step 3：执行清理

当计划明确后（或 Targeted 任务可直接在 Step 1 之后），使用下面提示词：

```text
执行下面的技术债清理。

清理任务：

<来自计划或定向描述的具体清理项>

基线：

<来自 Step 1 的验证命令、基线结果和覆盖缺口>

重要说明：

1. 在做任何修改前确认基线已记录。基线中可以包含既有失败，但必须明确记录，避免与当前清理引入的回归混淆。
2. 如果基线或早期调查显示任务比预期更广或更危险，先停下来，重新分类为 Scoped 或 Survey，再继续。
3. 一次执行一个清理项。
4. 每完成一项，都运行该项计划中的完整验证（测试、类型检查、lint、构建检查、使用情况检查、依赖专项检查、冒烟测试）。
5. 如果某项清理后验证失败：
   - 判断失败是回归（撤销或修复）还是既有问题（记录并继续）。
   - 在当前项清理干净之前，不要进入下一项。
6. 除非被明确要求，否则不要改变可观察行为。清理是移除浪费，而不是改变系统怎么工作。
7. 不要扩大范围。清理过程中发现新的债务，只记录到后续工作里，不要现在顺手修。如果新发现实质性改变了当前任务，先停下来重新规划，而不是边走边编。
8. 对于死代码移除：
   - 确认代码确实不可达（grep usages、检查动态引用、检查反射、检查配置驱动分发）。
   - 如果不确定，就标出来，不要删。
9. 对于依赖升级：
   - 检查 changelog 和迁移指南中的破坏性变化。
   - 每升级一个逻辑单元后就运行相关验证，不要等所有升级做完。
   - 如果依赖升级需要代码改动，要把这些改动和升级作为同一逻辑单元处理。
10. 对于 TODO / FIXME 处理：
   - 如果修复清晰且范围可控，就直接解决该 TODO。
   - 如果 TODO 描述的是更大任务，把它转成 issue 或记录下来，不要把大修复塞进清理流程。
11. 未经我明确许可，不要提交或推送。

结束时请提供：

1. 已完成的项
2. 跳过的项及原因
3. 变更文件
4. 每项之后的验证结果
5. 清理过程中发现的新债务（供后续跟踪，而不是当前会话继续处理）
6. 运行过的命令及结果
```

## Step 4：验证清理结果

执行完成后，或在 Step 3 修复问题之后，使用下面提示词：

```text
验证已经完成的技术债清理。

清理摘要：

<已清理的项、变更文件、每项之后的验证结果，以及跳过的项>

基线：

<来自 Step 1 的验证命令、基线结果和已知既有失败>

你的目标：

1. 重新运行工作流前面建立的完整验证集（Step 1 以及 Step 2 中计划的检查），然后与记录的基线对比。
   - 通过项不变且没有新增失败：很好。
   - 出现新增失败：调查是回归还是既有问题。
   - 当前跳过了原本在基线或计划中的检查：说明原因。
   - 删除了测试：逐个说明删除理由（例如它们是否只在测试已删除的死代码）。
2. 验证没有可观察行为变化：
   - 公共 API、接口和契约没有变化
   - 没有改变用户可见行为
   - 没有无意修改日志格式、指标名或事件结构
3. 验证此次清理在其声明范围内已经完整：
   - 是否还残留对已删除代码的引用？
   - 是否因为部分清理而引入新的不一致？
   - import、export 和配置是否都已与删除结果同步？
4. 检查清理遗留物：
   - 应该删除但还留着的空文件或空模块
   - 针对已删除代码的孤儿测试文件
   - 仍然引用已删除代码的陈旧注释
   - 残留的未使用 import
5. 如果验证结果表明任务比最初预期更广或更危险，明确指出，并建议在继续前重新规划。

未经我明确许可，不要提交或推送。

请提供：
- 与基线对比的验证结果（逐命令 before vs. after）
- 行为变化检查（none / list）
- 完整性检查（complete / incomplete items listed）
- 发现的清理遗留物（none / list）
- 总体结论：clean / needs follow-up
```

在没有未解决回归、行为变化、范围内未完成项或清理遗留物之前，反复执行 Step 3 与 Step 4。

推荐停止条件：

```text
如果 Step 4 发现回归、非预期行为变化、未完成清理，或仍应在当前范围内继续修的清理遗留物，就回到 Step 3 修复，再重新执行 Step 4。当记录的验证集相对于基线可以接受，且当前范围内的清理已经完成时，停止循环。
```

## Step 5：清理报告

当 Targeted 或 Scoped 清理任务在 Step 4 清理完毕后需要总结时，使用下面提示词：

```text
输出技术债清理报告。

清理目标：

<原始清理描述>

报告应包含：

1. 清理摘要（做了什么，为什么）
2. 已完成项，以及对应变更文件
3. 跳过的项及原因
4. 验证结果（基线 vs. 最终结果，覆盖测试、类型检查、lint、构建、使用情况检查和冒烟）
5. 可观察行为：确认未变化，或对例外情况给出理由
6. 清理过程中发现的新债务（供后续工作使用）
7. 按优先级排序的后续建议
8. 确认未执行提交或推送

未经我明确许可，不要提交或推送。
```

## 实务说明

- 另外也要遵循 [shared/workflow-conventions.md](shared/workflow-conventions.md) 中的共享约定。
- 技术债清理是范围蔓延最容易发生的地方。计划步骤的存在，就是为了划出边界。“既然都改到这里了，顺手再……” 这句话，往往会把 30 分钟清理变成 3 小时重构。
- 不是所有债都应该立刻偿还。有些 workaround 是承重结构；有些废弃模式虽然老，但稳定且低风险；有些 TODO 代表的是刻意延后的工作。评估步骤不仅要识别该清理什么，也要识别**现在不该清理什么**。
- 只有在没有任何引用时，死代码才真的是死代码。删除前要检查动态分发、反射、配置驱动路由、外部使用方和仅测试场景中的引用。
- 依赖升级不是免费的。主版本升级可能带来破坏性变化、行为差异或新的漏洞。把每个升级都当成独立单元，单独验证。
- 如果清理项最终需要设计决策，就停下来。设计决策属于功能开发工作流，不属于清理会话。
- 如果清理项最终变成了行为变更，就停下来。如果它是 bug，就走缺陷修复工作流；如果它是功能变更，就走功能开发工作流。
- 记录清理过程中发现的新债务。清理往往会暴露更多债务。把它们留给后续，而不是立刻展开。
- 对于仅评估、不改代码的任务，使用盘点报告（Survey Report）。不要把代码变更验证那套语言硬套到没有编辑的盘点任务上。
