# 项目初始化代理工作流

本文定义了一套可复用的代理工作流，用于从零启动新项目，包括从需求收集开始，到脚手架搭建、工具链设置、初始验证以及向后续开发交接。

该工作流面向为新仓库或绿地代码库搭建基础的 AI 编码代理。它适用于在技术栈、目录结构、构建工具、CI/CD、测试基础设施、依赖管理或部署目标方面存在实质决策的项目。对于极其简单的项目（单脚本、一次性原型），应跳过该工作流，直接搭建脚手架。

它强调需求清晰、技术选择有意识、构建可复现、尽早验证，以及避免未经授权的提交或推送。

## 预检环境检查

在进入分诊门之前，先收集目标环境的上下文：

```text
在为新项目搭建脚手架之前，先检查以下内容：

1. 项目应该准确放在哪个目标目录？
2. 目标目录或仓库是否已经存在？如果存在，记录当前状态。
3. 目标目录是否位于现有 git 仓库、monorepo、workspace 或组织模板内部？
4. 是否有适用于目标目录的本地说明，例如 AGENTS.md、CLAUDE.md、.editorconfig、CONTRIBUTING.md，或等效工作流文档？
5. 如果目标位于现有仓库中，当前分支和工作区状态是什么？
6. 是否存在与本项目无关、且绝对不能碰的未提交变更？
7. 当前机器上可用的语言运行时、包管理器和构建工具有哪些？
8. 是否存在组织级约束（必需的 lint、CI 提供方、许可证头、安全策略）？

如果目标目录缺失、有歧义，或意外地已经预填内容，先停下来提问，再创建文件。
不要覆盖、回退、删除或重新格式化与当前任务无关的用户变更。

请提供：

- 目标目录及其当前状态（空、尚不存在、或已预填）
- 父级工作区或模板约束（如果有）
- 适用的仓库本地说明
- 如果位于仓库内，当前分支 / 工作区状态
- 需要避免触碰的无关本地变更
- 可用工具链版本
- 已发现的组织约束
```

## 分诊门

开始之前，对项目进行分类，以选择合适的流程深度：

```text
将这个项目归类到以下类别之一：

1. Minimal：单一用途的脚本、CLI 工具或仅有一个入口点的库。没有服务端、数据库，也没有多服务协作。例子：实用 CLI、数据处理脚本、小型 npm 包、单文件 Lambda。

2. Standard：具有明确结构的应用或库。可能包含服务端、数据库、API 层、前端或构建流水线。例子：REST API 服务、Web 应用、SDK 或客户端库、微服务。

3. Complex：多组件系统、monorepo，或具有显著基础设施需求的项目。涉及多个服务、部署目标、数据存储、认证或跨团队集成。例子：前后端分离的全栈应用、多服务平台、需要基础设施即代码的项目。

根据分类：

- Minimal：使用下面的 Minimal Project Prompt，替代 Step 0 到 Step 5。直接用合理默认值搭脚手架。
- Standard：从 Step 0（收集需求）开始。项目计划（Step 1）可以省略不适用的章节。Step 2 和 Step 3 可以在同一会话中完成。
- Complex：完整执行 Step 0 到 Step 5。

项目描述：

<项目的简要描述>

请对这个项目分类，并建议应遵循哪些步骤。
```

## 全局规则

在每个提示词中使用 [标准编码规则](shared/safety-rules.md#standard-coding-rule)。

## Minimal Project Prompt

对于分诊门中归类为 `Minimal` 的项目，使用下面提示词替代 Step 0 到 Step 5：

```text
为下面这个最小项目搭建脚手架，不需要创建完整项目计划。

目标目录：

<项目目录的绝对路径或仓库相对路径>

项目描述：

<项目的简要描述>

需求：

<需求或验收标准（如果有）>

重要说明：

1. 创建文件前先确认准确目标目录。
2. 如果目标目录缺失、有歧义，或意外地已经预填内容，先停下来提问。
3. 在目标目录中，为该项目类型创建合理的目录结构。
4. 如果目标目录尚不在某个仓库中，则初始化版本控制（`git init`）。
5. 根据语言设置合适的包管理器和依赖文件（`package.json`、`pyproject.toml`、`go.mod`、`Cargo.toml` 等）。
6. 添加一个最小 README，包含项目名称、用途和运行方式。
7. 添加适合该语言和工具链的 `.gitignore`。
8. 如果该生态有强默认项，则包含一个基础 lint 或格式化配置（例如 ESLint、Black、rustfmt）。
9. 如果项目有明确入口点，创建一个可以成功运行的桩实现。
10. 运行所有可用验证（build、lint、type check），确认脚手架是干净的。
11. 如果你发现项目比预期更复杂，先停下来，建议切换到 Standard 或 Complex 工作流。
12. 未经我明确许可，不要提交或推送。
13. 不要覆盖、回退、删除或重新格式化无关的用户变更。

结束时请提供：

1. 创建的目录结构
2. 已配置的工具链和依赖
3. 运行项目的命令
4. 运行过的验证命令及结果
5. 建议的下一步
```

## Step 0：收集需求

在做技术和结构决策前，使用下面提示词澄清项目需求：

```text
审查下面的项目需求，并识别在搭脚手架前必须先做出的决策。

项目描述：

<项目概述>
<目标用户或消费者>
<关键功能或能力>
<约束或偏好>

你的目标：

1. 判断需求是否足够清晰，已经可以支持技术和结构决策。
2. 识别会影响以下内容的需求：
   - 语言和运行时选择
   - 框架选择
   - 数据存储需求
   - API 风格（REST、GraphQL、RPC 等）
   - 认证与授权
   - 部署目标（云厂商、容器、serverless、静态托管等）
   - CI/CD 需求
   - 测试策略
   - Monorepo 还是多仓库
3. 将结果拆分为：
   - Confirmed requirements
   - Inferred assumptions（附理由）
   - Open questions
4. 当答案会改变技术栈、项目结构或部署策略时，提出澄清问题。
5. 现在先不要开始搭脚手架。

未经我明确许可，不要提交或推送。

如果需求已经足够清晰，请明确说明，并给出 confirmed requirements、inferred assumptions 以及所有低风险 open questions。

如果需求还不够清晰，只问那些能解除项目规划阻塞的问题。
```

## Step 1：创建项目计划

当需求已经足够清晰后，使用下面提示词：

```text
根据下面的需求创建项目初始化计划。

将计划写成 Markdown，并放到 `reports/` 文件夹中，文件名使用：

`reports/<project-name>-init-plan-YYYY-MM-DD.md`

计划应包含：

1. 项目概览
2. 目标目录
   - 精确路径
   - 当前状态（不存在、空目录或已预填）
   - 父级仓库、workspace 或模板约束（如果有）
3. 已确认需求
4. 技术选型
   - 语言和运行时（含版本）
   - 框架（如果有，并说明理由）
   - 包管理器
   - 构建工具
   - Linter、formatter 和类型检查器
   - 测试框架
5. 目录结构
   - 计划中的目录布局，以及不明显选择的简要理由
6. 依赖清单
   - 生产依赖（名称、用途、版本约束）
   - 开发依赖（名称、用途、版本约束）
7. 配置文件
   - 要创建的配置文件列表及其用途（`tsconfig`、`eslint`、`dockerfile` 等）
8. CI/CD 设置（如果适用）
   - 流水线提供方和触发策略
   - 步骤（lint、test、build、deploy）
9. 环境与密钥管理（如果适用）
   - 环境变量策略（`.env`、secrets manager 等）
   - 示例 `.env` 模板（不能包含真实密钥）
10. 数据库 / 数据存储设置（如果适用）
   - Schema 初始化策略
   - 迁移工具
11. 认证 / 授权设置（如果适用）
12. 部署策略（如果适用）
    - 目标环境
    - 构建和部署步骤
    - 基础设施即代码（如果有）
13. 风险与开放问题
14. 初始化检查清单
15. 完成定义

省略所有不适用的“如果适用”章节。不要为了凑格式而填写占位内容。

需求：

<项目描述>
<已确认需求>
<约束条件>

如果某个技术或结构选择存在更好方案，请调整，并解释原因。

如果任何点不清晰或有风险，把它列到 “Open Questions（开放问题）” 下，并在做假设前先问我。

所有假设都要明确记录。不要把猜测当成需求。

该计划需要足够详细，能让另一位工程师在不额外补充上下文的情况下直接执行。

未经我明确许可，不要提交或推送。

现在开始。
```

### 建议的完成定义

把下面这一节加入每个项目计划中：

```markdown
## 完成定义

- 从脚手架工作树进行一次全新安装后，项目可以成功构建并运行。
- 所有已配置的验证命令都通过（lint、类型检查、构建、测试）。
- 目录结构与计划一致。
- 依赖已做合适的固定版本或版本约束。
- README 记录了安装、运行、测试和部署方式。
- `.gitignore` 覆盖了语言、工具链和 IDE 产物。
- 没有提交任何密钥或凭证。
- 如果适用，CI 流水线可以成功运行。
- 已记录已知限制和下一步工作。
```

## Step 2：评审项目计划

项目计划创建完成后，新开一个会话并使用下面提示词：

```text
项目初始化计划位于：

<path-to-plan>

请仔细评审这份计划。

你的目标：

1. 检查计划是否完整，是否已经可以执行。
2. 识别缺失的决策、风险较高的技术选择、依赖风险或初始化清单中的缺口。
3. 验证计划中的结构与工具链选择是否适合已陈述的需求。
4. 按严重级别分类每个发现：
   - Critical：脚手架开始前必须解决
   - Major：脚手架开始前应该解决
   - Minor：改进或建议
   - Question：需要澄清
5. 对于明显缺失的项目（例如没有列出 `.gitignore`），直接更新计划。
6. 对于技术取舍、架构决策或依赖偏好的选择，不要静默修改，把它们写成 “Review Findings（评审发现）”。
7. 如果你做了修改，新增或更新 “Review Findings（评审发现）” 章节，说明改了什么、为什么改。

每个发现都使用以下格式：

### [Severity] Finding title

- Location:
- Problem:
- Why it matters:
- Recommended fix:
- Status: Open / Fixed / Needs decision

现在先不要开始搭脚手架。
未经我明确许可，不要提交或推送。

如果计划已经可以执行，请明确说明，不要做无意义改动。
```

## Step 3：搭建项目脚手架

当计划稳定之后，使用下面提示词：

```text
已存在一份定稿的项目初始化计划，路径为：

<path-to-plan>

目标目录：

<项目目录的绝对路径或仓库相对路径>

按照计划开始为项目搭建脚手架。

重要说明：

1. 创建任何文件前，先仔细阅读计划。
2. 在提示词和计划之间确认准确目标目录，再开始创建文件。
3. 如果目标目录缺失、有歧义，或意外地已经预填内容，先停下来提问。
4. 如果目标目录尚不在仓库中，则初始化 git。
5. 按计划在目标目录中创建目录结构。
6. 初始化包管理器并安装依赖。
7. 创建计划中列出的所有配置文件。
8. 创建能够成功构建和运行的桩入口点或应用骨架。
9. 设置测试框架，并至少包含一个通过的示例测试。
10. 创建一个 README，包含项目名称、描述、前置要求、安装步骤、运行命令、测试命令和部署说明（如果适用）。
11. 创建一个 `.gitignore`，覆盖语言、框架、工具链、操作系统和 IDE 产物。
12. 如果计划中指定了 CI/CD，则完成对应配置。
13. 如果项目使用环境变量，则创建一个 `.env.example`（不能包含真实密钥）。
14. 运行所有可用验证：构建、lint、类型检查和测试。
15. 未经我明确许可，不要提交或推送。
16. 不要覆盖、回退、删除或重新格式化无关的用户变更。
17. 如果遇到计划未覆盖的决策点，先停下来提问，而不是猜。

结束时请提供：

1. 创建的目录结构
2. 已安装的依赖
3. 创建的配置文件
4. 运行过的验证命令及结果
5. 与计划的偏差（附理由）
6. 已知问题或未完成项
```

## Step 4：验证脚手架

脚手架搭建完成后，新开一个会话并使用下面提示词：

```text
一个新项目已经根据下面这份计划搭建完成：

<path-to-plan>

目标目录：

<项目目录的绝对路径或仓库相对路径>

验证该脚手架是否正确，并已准备好进入开发阶段。

你的目标：

1. 验证目录结构与计划一致。
2. 只删除已确认目标目录内的生成型安装/构建产物，然后从冷启动重新安装依赖，确认依赖可以干净安装。
3. 运行完整验证集：构建、lint、类型检查、测试。
4. 验证 README 中记录的安装、运行和测试说明可以按文档执行。
5. 检查 `.gitignore` 是否覆盖了合适的产物。
6. 检查脚手架中是否包含密钥、凭证或敏感数据。
7. 如果适用，验证 CI 配置在语法上是有效的。
8. 按严重级别分类每个发现：
   - Critical：开发开始前必须修复
   - Major：开发开始前应该修复
   - Minor：改进或清理
   - Question：需要澄清
9. 修复那些安全且明显在当前范围内的问题。
10. 未经我明确许可，不要提交或推送。
11. 不要覆盖、回退、删除或重新格式化无关的用户变更。

每个发现都使用以下格式：

### [Severity] Finding title

- Location:
- Problem:
- Why it matters:
- Recommended fix:
- Status: Open / Fixed / Needs decision

如果脚手架是干净的，请明确说明，并列出执行过的验证。
```

## Step 5：最终报告 / 交接

验证完成后，使用下面提示词：

```text
根据以下计划，为已搭建的项目输出最终初始化报告：

<path-to-plan>

目标目录：

<项目目录的绝对路径或仓库相对路径>

报告应包含：

1. 项目摘要
2. 技术栈（语言、框架、工具链、版本）
3. 目录结构
4. 依赖（生产和开发）
5. 已创建的配置文件
6. CI/CD 设置状态
7. 验证结果（构建、lint、类型检查、测试）
8. README 完整性
9. 已知限制或延期事项
10. 建议的下一步开发工作
11. 确认未执行提交或推送

另外还要验证：

1. 没有提交任何密钥或凭证。
2. 所有验证命令都通过，或失败已被记录说明。
3. 项目在从脚手架工作树做一次全新安装后可以构建并运行。

未经我明确许可，不要提交或推送。
```

## 实务说明

- 在适用时，也要遵循 [shared/workflow-conventions.md](shared/workflow-conventions.md) 中的共享约定。其中某些约定（例如基线、行为保持不变）更偏向已有代码库，对绿地项目未必适用。
- 整个流程中都以项目计划为事实来源。
- 除非需求明确要求，否则优先选择生态默认方案和被广泛采用的工具，而不是新奇或小众方案。
- 对依赖版本进行固定或约束，避免意外破坏。
- 不要提前安装当前还用不到的依赖。先保持精简，随着需求出现再增加。
- 把 open questions 视为阻塞项，除非答案能从明确需求中直接得出。
- 当项目准备好进入功能开发时，交接给 [feature-development-agent-workflow.md](feature-development-agent-workflow.md) 去实现第一个功能。
